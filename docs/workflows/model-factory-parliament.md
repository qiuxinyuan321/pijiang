# 皮匠议会工作流

## 目标

把“讨论一下 / 开个会 / 商量一下”这类需求，变成一套真实、可核验、可重复执行的多模型议会流程。

皮匠的核心前提固定为：

- 不是单模型多角色扮演
- 而是多个真实模型位分别承担分析职责
- 最终做的是发散、对抗、整合与收敛

## 标准议会

官方默认标准配置固定为 `11 席公开议会（10 路变体 + 1 席 fusion）`：

1. 主控
2. 规划者
3. 外部搜索者 1
4. 外部搜索者 2
5. 裨将 1 / Kimi
6. 裨将 2 / GLM-5
7. 裨将 3 / MiniMax
8. 裨将 4 / Qwen
9. 混沌者
10. 质疑者
11. 融合者

## 首发命令

公开命令固定为：

```powershell
cpj init
cpj doctor
cpj demo
cpj integrate <host>
cpj run --brief "<brief.md>" --topic "<议题>"
```

## 新用户顺序

不要一上来就直接真实运行。

推荐顺序固定为：

1. `cpj init`
2. `cpj doctor`
3. `cpj demo`
4. 配置真实 provider
5. `cpj run`

## 首次运行规则

首次 `cpj run` 前必须：

1. 先解释工作原理
2. 先提醒多模型决策会更慢
3. 用户确认之后才真正发起调用

## 进度反馈

运行期间必须持续给反馈，不允许黑盒等待。

最少要能看到：

- 当前阶段
- 当前席位状态
- 已完成席位数 / 失败数
- 已生成产物

## readiness gate

`cpj doctor` 和 `cpj run` 之间必须有正式门禁。

- `ready`：可直接真实运行
- `warning`：需要 `--allow-degraded` 才能继续
- `blocker`：默认拒绝真实运行

`cpj demo` 不依赖真实 API，用于先验证安装、模板和产物链。

## Obsidian

Obsidian 是强推荐默认体验，不是硬阻断依赖。

启用后，官方 Vault 模板应能看到：

- 当前议题总览
- 11 席议会拓扑
- 外部搜索位
- 裨将层
- idea map
- fusion decisions
- final draft
- 运行历史
- 执行进度

## 原则

1. 诚实优先于漂亮结果
2. 卡住就报卡点，不绕成假结果
3. 真实运行结果优先于先验判断
4. 不把“模型扮演角色”描述成系统原理
